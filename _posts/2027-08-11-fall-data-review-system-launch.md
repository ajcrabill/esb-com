**Launching the Fall Data Review System — Before the First Data Review, the Board Needs to Know What It Is Looking At**

August 11, 2027

---

In July, the board redesigned the fall data review system. The spring data review cycle was reviewed. The fall metric set was designed. The data format was selected. The fall data review schedule was confirmed. The decision-recording protocol was designed.

That was the design phase. Design is necessary. Design is not sufficient.

Now it is August. The system needs to be launched. And launching has a specific meaning in this context: the board needs to confirm that every component of the data review system is operational before the first fall data review, which is likely in October.

**What a launched data review system looks like**

A launched data review system has six confirmable components. If any of these components is not confirmed, the system is not launched — it is designed but not operational.

**One: The metric set is finalized and shared with the superintendent.**

The board designed the fall metric set in July — the specific metrics that will tell the board whether progress is being made against the goals adopted in January. The metric set needs to be finalized in August and shared with the superintendent in writing.

The reason this matters: the superintendent needs to know, in advance, what data the board expects to see at each data review. If the board has not communicated the metric set, the superintendent is preparing data based on assumptions. And assumptions about what a board wants to see are often wrong.

The metric set should include, for each metric: the metric name, the data source, the baseline value, the target value, and the data frequency. A metric set that includes these five elements is a governance instrument. A metric set that includes only the metric name is a wish list.

**Two: The data format is confirmed with the data provider.**

The board selected a data format in July — one-page scorecard, dashboard, or data packet. The data format needs to be confirmed in August with the person or team responsible for producing it.

The reason this matters: a one-page scorecard requires a different data preparation process than a dashboard or a data packet. If the board selected a dashboard format but the superintendent's data team is preparing a data packet, the board will receive the right data in the wrong format. And the wrong format reduces the board's ability to make decisions from the data.

**Three: The data review schedule is on the governance calendar.**

The fall data review schedule was confirmed in July — October and January as the minimum, quarterly if possible. Those dates need to be on the board's published governance calendar by the end of August.

The reason this matters: if the data review dates are not on the calendar, they will be displaced by other priorities. School districts are busy. The superintendent's schedule fills up. Board members' schedules fill up. A data review date that is not on the calendar by August will not be on the calendar in October.

**Four: The data review preparation responsibilities are assigned.**

Every data review requires preparation work: the data packet needs to be prepared, the board packet needs to be distributed in advance, the facilitation needs to be designed, and the decision-recording format needs to be ready.

Those responsibilities need to be assigned to specific people by the end of August. Not "the superintendent will prepare the data" — that is vague. "The superintendent's data team will prepare the one-page scorecard and deliver it to the board president by October 1. The board president will distribute the scorecard to the full board by October 3. The board president will facilitate the data review at the October 10 board meeting. The board secretary will record the decisions in the board minutes and in the goal-tracking document."

Specific people. Specific dates. Specific deliverables.

**Five: The decision-recording protocol is tested.**

The board designed a decision-recording protocol in July — the format for documenting what the board decides after reviewing data. The protocol needs to be tested in August, ideally with a practice run using the Q2 data that the board has already reviewed.

The reason this matters: a decision-recording protocol that looks good on paper may not work in practice. The board may find that the format is too detailed, or not detailed enough, or that the recording process takes too long, or that the decisions are not actionable. August is the time to discover those issues — not October, when the first real data review requires a functioning protocol.

**Six: The board's data literacy is assessed.**

Most boards adopt data review systems that are more sophisticated than the board's current data literacy level. The board designed the metric set, the data format, and the review schedule based on where the board wants to be. But the board's ability to interpret, question, and make decisions from the data depends on where the board is right now.

The board should assess its data literacy in August — not with a formal assessment, but with a honest conversation during the August board work session. Questions to ask: does every board member understand how to read the one-page scorecard? Does every board member know what each metric measures, where the data comes from, and what the targets mean? Does every board member feel confident asking questions about the data during a public board meeting?

If the answer to any of these questions is no, the board should schedule a data literacy training session before the first data review. The training does not need to be elaborate. A 60-minute work session where the superintendent walks the board through a sample data packet and the board practices asking questions and making decisions is sufficient.

**What this prevents**

The most common reason boards stop doing data reviews after the first one is not that the data reviews are not valuable. It is that the board was not prepared for the first one. The data was in a format the board could not read. The metric set was unclear. The preparation was incomplete. The board meeting ran long because the data review took twice as long as expected.

The board that launches the data review system in August — confirms every component, assigns every responsibility, and tests every protocol — enters the fall knowing that the first data review will work. And a first data review that works becomes a habit. A first data review that does not work becomes a reason to stop doing data reviews.
